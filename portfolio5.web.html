THE TREACHEROUS COMPLEXITIES

In order to understand the need for such a niche application, one must first review the treacherous complecities of gymnastics. Today, the difficulty of the routine and the execution (how well the skills are performed) are combined to create the final score:

The difficulty ("D") score starts at 0.0 and increases with every difficult skill performed. The execution ("E") score begins at a 10.0, and the judges deduct for errors in performance such as a fall off the apparatus or a step on the landing of a dismount. In this new system, there is theoretically no limit to the score a gymnast can achieve. The top performances in men's gymnastics right now are receiving scores in the 15s and, occasionally, the low 16s.

The D SCORE

SKILL VALUE: 10 skills count to your start value. No repeating skills. An alert modal will notify the user if he attempts to repeat a skill, or add more than 10 skills. Each skill is valued A-H.

A = +0.1

B = +0.2

C = +0.3

D = +0.4

E = +0.5

F = +0.6

G = +0.7

H = +0.8

REQUIREMENT GROUPS: Each skill falls into a requirement group, You cannot count more than 5 skills in each requirement group. For each requirement group completed, your routine is awarded +0.5 once. Group I: Non-Acrobatic Skills is one of the requirement groups. This group includes, strength elements, Pommel Horse elements, and dance elements. Only 2 strength elements and only 2 pommel elements are allowed per routine. A modal will alert the user if he attempts to add more than 2 of the above element groups within Group I or if they have attempted to include more than 5 skills of a particular requirement group.. The other groups include Group II: Front Flipping Skills, and Group III: Back Flipping Skills.

The Group IV requirement: Dismount, is especially complex. If you perform a 10th skill from Group II or Group III, you are awarded points for a Group 4 requirement. However,if your 10th skill is an A or B skill, you are awarded +0. If your 10th skill is a C skill, you are awarded +0.3 instead of +.05. If your 10th skill is a D or higher skill, you are awarded the full +0.5.

DOUBLE FLIPPING REQUIREMENT: If a gymnast does not include a double flipping skill, yhe i s deducted -0.3. If he does include a double flipping skill, he is not rewarded or deducted (+0.0).

SKILL NUMBER POINTS: If you have 1 skill in your routine, you are rewarded +2.0. If you have 3 skill in your routine, you are rewarded an additional +2.0 (+4.0). If you have 5 skill in your routine, you are rewarded an additional +2.0 (+6.0). If you have 7 skill in your routine, you are rewarded an additional +4.0 (+10.0)

CONNECTION BONUS: If a B or C skill is performed in conjunction with a D or higher skill, you are awarded an additional +0.1. If a D or higher skill is performed in conjunction with a D or higher skill, you are awarded an additional +0.2

START VALUE= skill values + requirement points + skill number points + connection bonus.

NAVIGATION

Upon opening the page you are presented with 3 categories of skills organized by its requirement group. Within each category are sub-categories organized by thier value. Within the value category the user can choose a skill. Although the organization of skills within value categories within requirement groups is the best way to organize for skill selection, the full screen accordian style selection section is an eye sore. Perhaps using checkboxes to filter requirement groups and skill values may have been a prettier way to navigate.

SELECTING A SKILL

After clicking on a value link, the user is presented with a variety of skills. Each skill is wrapped neatly in a gradient box including its skill name, description, and illustration. I reiterate its skill value, and requirement group. I also include a video link that opens a new tab showing the user how the skill is performed. Each skill box was designed using CSS3 and the bootstrap grid. If had I more time, I would have the video pop up on a modal to avoid moving to a separate tab. I would also create a fixed length for each description so that each skill box matched in length.

SELECTING A SKILL

Upon selection, a skill will be saved to the users database which is organized using Mongoose and sessions. A more in depth graphic for the choosen skill will pop up at the top of the user's screen. The new skill box includes the skill name and illustration. I also included every element of the skill that will earn points. For example, I indicated that the first skill is at value "H" which earns 0.8. I showed that it is the first skill, which earns 2.0. I also highlight that requirement group III and the "double" requirement have been completed earning .05. Completed requirement groups are highlighted in purple informing the user of incomplete requirement groups. The gradients are used to represent the skills value so that when a user is looking at a full routine, he can identify where the routine can improve in difficulty. Upon hovering over the skill name and requirement groups, a description is offered in a contrasting black container. After choosing ten skills a modal presents the user with instructions to choose which skills are available for connection by directing them to the animated "connection checkbox". Clicking on the checkbox will add 0.1 or 0.2 depending on the difficulty of the connection. Finally, I have included a video link and delete option. If I had more time, I would message the user that a new skill has been choosen with a self-closing and fading alert. I would also play around with more color options including the white text which is not etirely user friendly.

MINIMUM VIABLE PRODUCT

Calculating a gymnastics routine start value is tedious, time consuming, and ridled with opportunity for user error. At GymnasticsStartValue.com, a gymnast can avoid mistakes and frustration by letting the advanced algorythms do the work. It is the first ever actionable list of gymnastics skills with video links on the internet. My priority for this site was to create advanced algorythms to solve treacherous calculations. I would focus heavily on design if I was given more than 10 days to complete the project. Although I like the horizontal row of skills, It does not allow for a fully responsive design. The site is only functionable using a minimum width of 1024px. I beleive that a typical user is not used to reading information in the format presented. Rather than re-invent the wheel, I would like to present the information closer to that of an excel spread sheet. The colors are pleasing, however, separating skill values using more contrasting color could help the user to differentiate skills and difficulty throughout the routine. Navigating through skills is efficient, but ugly, and the presentation of the actual start value should be more exciting. Finally, I would have liked to reduce the amount of scrolling throughout the site. I believe it would be more managable to search for skills while the list of choosen skills is fully visible.

LOW-FIDELITY WIREFRAME

My intial goals were to incorporate an "excel-like" set of rows and columns, and for the user to read their routine and choose skills without scrolling. While sketching an interface with these goals in mind, I discovered several redunancies and design opportunities. While choosing a skill it would be silly to incorporate dropdown menus for requirement groups and and values. Simply using highlighted buttons would prevent the extra click. Also, a user will likely only choose gender, level, and event once, and likely to choose requirement groups and values several times. Separating the "once-chosen" and "multiple-choosen" values will prevent confusion. I discovered that showcasing completed requirements per skill would be redundant so I implemented a completed requirements section so that they would be displayed clearly only once.

WIREFRAME EXPIREMENTS

Next, I experimented with some layouts by creating simple wireframes using adobe illustrator. I knew that I wanted to split the screen into two major rows or colums. I learned that it would be best to separate all information regarding the skill search from the routine presentation.

FUNCTIONALITY THEN FUN

I feel the most important element of a user's experience is to easily navigate the site intuitively, but I also wanted to create a sense of joy and excitement. I thought it would be fun to present the user's start value in a manner in which they are accustomed. If a users start value exceeds a 14.0 they are presented with a bronze medal. Above a 15.0 earns a silver, and above a 16.0 earns a gold. I used a flat design incorporating gymnastics' most iconic skill, the iron cross. However, I'd like to incorporate a series of medals that correspond to each event. That way, the user is motivated to collect all six gold medals.

HI-FIDELITY WIREFRAME

The user is presented with a navbar showcasing the "once-choosen" selections. After choosing their gender, level (junior/ elite), and event they will not have to return to the navbar. The right side of the screen is the "skill selection section." I choose to put this on the left side of the screen because this where the eye is drawn to initially. A heading labeled "skills" directs the user to the selection section where a search bar is placed in a familiar space. Next, he or she is promted to chose a group and value for a particular set of skill. The highlighted button seems to be the most efficient way of selecting categories. The user can scroll through the list of skills represented with a picture, name, description, value, group, and video. The presentation of skills remains consitent with the display of the routine section. Users can select their skill underneath it's name or drag it to the appropriate slot in the routine section.


The routine section shows the skill number, video button, delete button, name, value, group, group value, connect option, connect bonus, and total points per skill. I also included highlighted stars to call attention to a particular skill. The blue plus button offers an accordian style drop down for notes. The elements of each skill are easily readable and "excel-like." Although there was no room to label the buttons, I feel their design conveys their function, and hovering over the button will describe its function. I felt the video button and delete button would be used more, so I placed them on the left side of the section. When the buttons are all placed in-line, it is overwhelming to the eye. I rarely use red throughout the site. When there is an opportunity to improve your routine simply, I use the red warning label. Below the warning label is a list of requirements for a routine. When a skill fullfills a requirement the text turns from red to green. The last requirement is undetectable by the computer so it remains black. When a user clicks on a requirement, the set of skills that full-fill that requirement will appear in the skills section. Finally, the medal is presented when 10 skills are chosen inspiring user excitement. Clicking "Save Routine" will send the ten skills to the database and can be reviewed in the saved routine section of the site accessible in the navbar.

HOTEL COMPARE
After 2 months of programming bootcamp I was assigned to create my first CRUD (Create, Read, Update, Delete) application within 5 days. Hotel Compare is an app used to find hotels by location. My goal in this project was to utilize my understanding of back-end software development specifically by incorporating an Application Programming Interface. For Hotel Compare, I used the Google Maps and Google Places API. Although some elements of aesthetic design were incorporated, the purpose of this application was to test my back-end development skills.

INPUT

The user is introduced with an image of a luxurious bed amongst adventerous mountains, conveying a message that one can vacation anywhere. The juxtoposition of comfort and excitement projects the novelties that users are looking for in a vacation. Below the image is an input-box instructing the user to "enter a city." Upon entering the first character, the Google Maps API will trigger an auto-complete which assumes the destination location. After submitting a city or directly clicking on the map, the window will automatically scroll down featuring the map and hotel options. Although the auto-scroll is conveneint, the design would be better suited to minimize scrolling.

CHOOSE A HOTEL

Submitting a city will trigger the Google Places API which outputs 20 hotel choices within a 5 mile radius of the city's center. Clicking on a hotel within the accordian-style collapse toggle will offer the user information regarding the hotel's name, address, phone number, rating, reviews and website link. If the hotel seems promsing, the user can save it to their favorites. A post request is sent to the server, saved to the database, and presented below the map.

MINIMUM VIABLE PRODUCT

While the site successfully uses Google APIs to search for hotels by location, the design is lacking. First, there is too much scrolling and content for one page. Id prefer to have the "favorite" hotels saved and displayed on a seperate page. The design is unappealing and not trustworthy. The attributes for a chosen hotel are held to minimum and visually unappealing.
